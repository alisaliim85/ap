# تقرير تدقيق صلاحيات وأدوار النظام

## 1. نظرة عامة (Overview)
يقدم هذا المستند جرداً شاملاً لجميع أدوار المستخدمين (Roles)، المجموعات (Groups)، والصلاحيات (Permissions) سواء القياسية أو المخصصة التي تم تحديدها في كود مشروع **AP PLUS**.

## 2. مسرد المصطلحات (Glossary)
لتوضيح الفرق بين أنواع الصلاحيات المستخدمة في التقرير:

*   **الصلاحيات القياسية (Standard Permissions):**
    هي الصلاحيات التي يقوم نظام Django بإنشائها **تلقائياً** بمجرد تعريف أي جدول (Model) في قاعدة البيانات. وتغطي العمليات الأربع الأساسية:
    *   **Add**: صلاحية إضافة سجل جديد (مثلاً: `add_member`).
    *   **Change**: صلاحية تعديل بيانات سجل موجود (مثلاً: `change_member`).
    *   **Delete**: صلاحية حذف سجل (مثلاً: `delete_member`).
    *   **View**: صلاحية الاطلاع على التفاصيل فقط (مثلاً: `view_member`).
    *   *الاستخدام:* يشير التقرير إليها بـ "صلاحيات قياسية مستخدمة" عندما يستخدم الكود هذه الصلاحيات الجاهزة.

*   **الصلاحيات المخصصة (Custom Permissions):**
    هي صلاحيات **إضافية** قام المبرمج بتعريفها يدوياً في كلاس `Meta` للجدول. وهي تستخدم للتحكم في إجراءات العمل الخاصة (Workflows) أو الدخول لشاشات محددة.
    *   *أمثلة:* `can_approve_hr` (عملية اعتماد)، `view_broker_dashboard` (دخول شاشة).

## 3. أدوار المستخدمين (User Roles)
يحدد النظام الأدوار التالية في `accounts.models.User.Roles`. تحدد هذه الأدوار الوظيفة الأساسية للمستخدم في النظام.

| رمز الدور | الاسم الظاهر | الوصف |
| :--- | :--- | :--- |
| `SUPER_ADMIN` | مدير النظام (المالك) | صلاحيات كاملة للنظام (Superuser). |
| `BROKER_ADMIN` | مدير الوسيط | إدارة النظام لشركة الوساطة. |
| `BROKER_STAFF` | موظف وسيط | الطاقم التشغيلي لشركة الوساطة. |
| `HR_ADMIN` | مدير موارد بشرية | مسؤول النظام لدى الشركة المتعاقدة (العميل). |
| `HR_STAFF` | موظف موارد بشرية | موظف الموارد البشرية لدى العميل. |
| `PHARMACIST` | صيدلي | موظف لدى الشريك الطبي (صيدلية). |
| `CHRONIC_ADMIN` | مدير رعاية مزمنة | مدير مركز رعاية الأمراض المزمنة. |
| `CHRONIC_STAFF` | كادر طبي | أطباء/ممرضين للرعاية المزمنة. |
| `VIEWER` | مدقق / مشاهد | صلاحية قراءة فقط (للتدقيق). |
| `MEMBER` | عضو / مستفيد | المستخدم النهائي (الموظف وعائلته). |

## 4. استراتيجية المجموعات (Groups Strategy)
يستخدم النظام استراتيجية **ربط 1-إلى-1** بين **الأدوار (Roles)** و **مجموعات دجانغو (Django Groups)**.
- **المنطق:** محدد في `accounts.signals.move_user_to_group`.
- **السلوك:** عندما يتم تعيين `role` لمستخدم (مثلاً `HR_ADMIN`)، يقوم النظام تلقائياً بإضافته لمجموعة دجانغو بنفس الاسم (`HR_ADMIN`) وإزالته من المجموعات الأخرى.
- **الوضع الحالي:** يتم إنشاء المجموعات تلقائياً، ولكن **لم يتم إسناد الصلاحيات لهذه المجموعات برمجياً**. يجب إسنادها يدوياً من لوحة التحكم أو عبر سكربت (Fixture).

## 5. منطق سير العمل والإعدادات (Workflow Logic)
بعض الصلاحيات أو الإجراءات مشروطة بـ **إعدادات العميل** وليس فقط بصلاحيات المستخدم.

*   **سير عمل المطالبات - مراجعة الـ HR:**
    - **المنطق:** في `claims/models.py`، الانتقال `submit_direct_to_broker` يفحص الشرط `can_bypass_hr`.
    - **الإعداد:** يتم التحكم بهذا عبر إعدادات العميل: `client.get_claim_setting('bypass_hr_review', False)`.
    - **الأثر:** إذا كانت قيمة `bypass_hr_review` تساوي **True** (تجاوز)، فإن المطالبة تقفز عن حالة `SUBMITTED_TO_HR` وتذهب مباشرة إلى `SUBMITTED_TO_BROKER`، وبالتالي يتم تجاوز الحاجة لصلاحية `can_approve_hr` لهذه المطالبة تحديداً.

## 7. نتائج التدقيق والتوصيات
1.  **صلاحية مفقودة التعريف:** الصلاحية `accounts.manage_company_staff` مستخدمة فعلياً في `accounts/views.py` (لإدارة موظفي الـ HR) وفي القائمة الجانبية، لكنها **غير معرفة** في `accounts.models.User.Meta`.
    - *الأثر:* فحص هذه الصلاحية (`user.has_perm`) سيعود دائماً بـ `False`، مما قد يمنع مدراء الـ HR من إدارة موظفيهم.
    - *الإجراء المطلوب:* إضافة `("manage_company_staff", "Can manage company staff")` إلى `User.Meta.permissions`.

2.  **عدم الإسناد البرمجي:** بينما يتم إنشاء "مجموعات" (Groups) لكل دور، لا يتم إسناد الصلاحيات لهذه المجموعات في الكود.
    - *الإجراء المطلوب:* إنشاء أمر إداري (Management Command) أو ملف (Fixture) لربط الصلاحيات المذكورة أعلاه بمجموعاتها المناسبة (مثلاً: إسناد `manage_members` لمجموعة `HR_ADMIN`).
